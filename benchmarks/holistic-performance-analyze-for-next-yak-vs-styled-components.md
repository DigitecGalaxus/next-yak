
### EXP-20260609-02 — Agent A: NestedComponents autopsy (refs: H7, H12, backlog #1/#2)
- Env: same as EXP-01; unminified next-yak dist; 5000 iterations/case; line-level positionTicks from cpuprofile
- Setup: `pnpm bench:profile nested-{yak,styled,vanilla} 5000` + heap profiles; analysis vs `dist/internal.js` (unminified) and the compiled fixture
- Numbers: nested-yak 1038 ops/s / nested-styled 2307 / nested-vanilla 7249. CPU split: yak-runtime **79.2%**, react-dom 11.7%, gc **1.2%**. Inside the `Yak` closure (75.2% of process): `new Set(className.split(" "))` **53.3%**, `Array.from(set).join(" ")` **17.8%**, theme rest-spread clone 13.2%, `{theme,...props}` spread 7.8%, `useTheme()` 2.5%, re-entry/createElement 2.0%. Per `<div>` with a 5-level chain: Yak executes **6×** → ≈18 object clones, 6 Sets, 6 useTheme calls, 11 split/join round-trips — for a 100% static chain whose class list is computed once and then redundantly re-split/re-joined 5 more times.
- **Discovered defect:** `styled.tsx` gates `useTheme()` on `runtimeStylesFn.length` — that's function **arity** (the static processor has arity 2), so the gate is always truthy and `useTheme()` runs for every component including fully static ones. Cheap in SSR (2.5%) but client-side it subscribes every styled component to theme context (feeds H8/D6).
- Verdict: H7 🟡 PARTIAL (per-level prop cloning real at ~21% of Yak time, but no attrs chains involved and GC is negligible — the named mechanism was wrong, the multiplier is right); H12 🟡 PARTIAL (×6 re-entry confirmed as the structural multiplier; the marker props themselves cost 1–2% and the processed-guard actually saves work). The dominant cost is CPU marshaling (Set/split/join), not allocation pressure.
- Next action: P1 (Set→string, kills 71% of Yak time here), P5 (flatten chain, kills 5/6 executions), P3 (static fast path incl. fixing the arity-vs-dynamic gate). P4/P6 demoted (≤2% each).
